# Code Review Flow — base rate (2026-07)

The measured quality of the **increment 1–3** grounded context (facts only: file
set, changed files + per-file churn, git state). Generated by `nix run
.#review-eval` over a code-heavy dual-language corpus; assessed on the
[rubric](README.md#the-rubric-shared-by-both-judges) by the assistant and (when an
endpoint is supplied) GLM-5.2. This is the number future increments improve on.

## Corpus

| Item | Lang | Source | Changed files |
|---|---|---|---|
| `rust-pr-{84,87,88,94,95,96,98,99}` | Rust | this repo (worktree per head commit) | 6–28 |
| `go-s3-secret-file` (xtcp2 #56) | Go | flake-pinned trees | 2 |
| `go-ns-churn-thread-leak` (xtcp2 #52) | Go | flake-pinned trees | 8 |

Rust = the 8 most-recent code-heavy merge PRs (≥3 `.rs`, Rust-dominant). Go =
the two hash-pinned xtcp2 changes. Samples in [`samples/`](samples/).

## Deterministic results

| Metric | Result |
|---|---|
| Contexts generated | **10 / 10 complete** (`repo-change=ok`) |
| Language detected | **10 / 10 correct** (8 rust, 2 go — via `Cargo.toml`/`go.mod`) |
| Host / relationship | **10 / 10** `github` · `clone` (correct) |
| Changed-file set | matches `git diff --name-only` for every item |
| Tracked-file count | faithful per commit (e.g. pr-95 = 403, pr-98 = 586, go = ~2150) |
| Collection latency | mean **75 ms**, range 44–117 ms |

The deterministic layer is, as designed, **exactly correct** — the facts are read
from git, so accuracy is not really in question; latency is negligible.

## Assessment — the assistant

Scores 1–5 (see rubric). These are aggregate over the corpus; per-language notes
follow.

| Dimension | Score | Rationale |
|---|:--:|---|
| **Groundedness / accuracy** | **5** | Every stated fact is verifiable against git and correct. No hallucination is structurally possible — the context is tool-derived. Language, host, relationship, per-file additions/deletions all check out, for both Rust and Go. |
| **Review-readiness / completeness** | **2** | The context orients a reviewer to the change's *shape* — which files, how much churn each, the repo's identity — but supplies nothing to actually *review*: no diff/hunk content, no statement of what the change does, no static-analysis findings, no blast-radius. You learn **where** to look, not **what** changed or whether it is right. |
| **Signal-to-noise** | **5** | Concise and entirely relevant; every line is a fact a reviewer wants, the file list is bounded, and the collection accounting is a useful footer. No filler. |

**Rust vs Go:** no quality difference. The worktree (Rust) and reconstructed
pinned-tree (Go) paths produce identically faithful bundles; `go.mod` vs
`Cargo.toml` detection both land. A larger change (pr-98, 28 files) is as accurate
as a 2-file one — the context scales cleanly.

**The dominant gap, concretely.** Reviewing `rust-pr-95` from its context, I know
`agent.rs` changed by +118/-60 and `verifier_e2e.rs` by +188/-9 — but I cannot see
*a single line* of what changed, nor read that "enforce mode blocks a Revise/Deny
call and feeds the message back." The context is a **table of contents without the
chapters**. That is the honest ceiling of a facts-only bundle.

### Gaps → the roadmap that closes them

| Gap a reviewer still needs | Addressed by |
|---|---|
| **The actual diff / per-file hunks** | *not yet an increment* — `ChangeSet` already carries `patch`; `render_facts` omits it. **Cheapest, highest-value next add.** |
| **What the change does (summary/intent)** | increment **8** (cheap-LLM before/after summaries) |
| **Correctness signals (lint/vet/clippy)** | increment **5** (static analysis) |
| **Blast radius (who calls the changed code)** | increment **6** (AST / call-graph) |
| **House-style conformance** | increment **7** (style fingerprint) |
| commit message / PR intent | minor; a `RepoBackend::log` line would add it |

**Conclusion.** The increment 1–3 context is **accurate (5) and clean (5) but thin
(2)** for actually reviewing code. The base rate to beat is
**readiness ≈ 2 / 5**. The single biggest lift is showing the diff content, then
the change summary (incr 8) and static-analysis findings (incr 5). Re-run this eval
after each increment; readiness should climb while groundedness stays at 5 (any
drop means a later increment started *guessing*, which the whole design forbids).

## Assessment — GLM-5.2

Produced by `nix run .#review-eval -- --judge` over the same 10 contexts (model
`/model`, one `{groundedness,readiness,signal,gaps}` verdict each; two samples in
[`samples/judge/`](samples/judge/)). Aggregate:

| Dimension | Mean | Spread |
|---|:--:|---|
| **Groundedness / accuracy** | **5.0** | 5 on all 10 |
| **Review-readiness / completeness** | **2.1** | 2 on nine, 3 on one (a 22-file PR) |
| **Signal-to-noise** | **4.5** | 5 on the smaller changes, 4 on the larger (a 28-file listing reads slightly noisier) |

GLM's most-cited gaps, across the corpus:

1. **The actual code diffs / hunks** (≈8 mentions) — "I can see 7 files changed but not a single line."
2. **Commit message / PR description explaining intent** (≈6 mentions) — the *why*.
3. Test/build status, a semantic summary of what changed, and rationale.

## Comparison + conclusion

The two judges **converge almost exactly**:

| | Assistant | GLM-5.2 |
|---|:--:|:--:|
| Groundedness | 5 | 5.0 |
| Review-readiness | 2 | 2.1 |
| Signal-to-noise | 5 | 4.5 |

No divergence to caution about — GLM did **not** inflate readiness by inferring
intent from filenames; it explicitly docked the score *because* the intent and the
diff are absent. Both judges independently name the **same top-two gaps in the same
order: show the diff content, and surface the commit message / intent.**

That agreement makes the base rate and the next step unusually well-founded:

- **Base rate to beat: groundedness 5, readiness ≈ 2, signal ≈ 4.5–5.**
- **Do first (both judges' #1–2):** render the diff hunks (already in
  `ChangeSet.patch`, merely dropped by `render_facts`) and add the range's commit
  messages (`RepoBackend::log base..head`). Neither needs a model; both move
  readiness directly.
- **Then:** change summaries (incr 8) and static-analysis findings (incr 5) for the
  deeper "what/why/is-it-correct" signal; a churn/history collector for hot-spot +
  line-age context.

Re-run `--judge` after each increment. **Readiness should climb while groundedness
holds at 5** — if groundedness ever drops, a later increment started *guessing*,
which the whole design forbids.
